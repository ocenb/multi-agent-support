# Multi-Agent Support System — Product & Technical Specification (MVP)

## 1. Vision и цель проекта

**Название:** Multi-Agent Support System  
**Формат:** портфолио-проект (production-style MVP)  
**Цель:** спроектировать и реализовать многоагентную ИИ-систему, которая автоматизирует L1-поддержку для e-commerce и IT-сервиса.

### 1.1 Бизнес-ценность

Система должна:

1. Снижать нагрузку на первую линию поддержки за счет автообработки типовых запросов.
2. Ускорять получение ответа пользователем.
3. Повышать качество маршрутизации инцидентов (включая баг-репорты).

### 1.2 Основная гипотеза MVP

Если разделить поток обращений на специализированные агентные ветки (`INFO`, `STATUS`, `BUG`), то доля обращений, обработанных без участия человека, вырастет при контролируемом уровне ошибок.

## 2. Scope проекта

### 2.1 In Scope (MVP)

1. Классификация входящих запросов на `INFO`, `STATUS`, `BUG`.
2. RAG-ответы по базе знаний для ветки `INFO`.
3. Tool Calling к Mock API для проверки статуса заказа в ветке `STATUS`.
4. Триаж багов и формирование структуры тикета в ветке `BUG`.
5. Базовая наблюдаемость: трассировка шагов, логи, причины маршрутизации.

### 2.2 Out of Scope (не делаем в MVP)

1. Полноценный UI личного кабинета оператора.
2. Поддержка мультиязычности.
3. Автоматическое обучение/переобучение моделей в проде.
4. Интеграция с enterprise IAM/SSO.
5. Полноценные SLA и HA-кластеризация.

## 3. Целевые пользователи и сценарии

### 3.1 Персоны

1. **Клиент магазина/сервиса:** хочет быстро получить ответ.
2. **L1-оператор:** хочет меньше рутины и корректную эскалацию.
3. **Разработчик/тимлид:** хочет качественные баг-тикеты с контекстом.

### 3.2 User Stories

1. Как клиент, я хочу узнать правила возврата и доставки, чтобы не ждать оператора.
2. Как клиент, я хочу узнать статус заказа по номеру.
3. Как клиент, я хочу сообщить о баге и быть уверенным, что обращение дойдет до команды.
4. Как оператор, я хочу видеть, почему запрос попал в конкретную ветку.

## 4. Технологический стек

1. **Оркестрация:** Langflow.
2. **LLM:** модель с API-доступом (например, OpenAI / Groq / GigaChat / YandexGPT).
3. **RAG-хранилище:** ChromaDB или FAISS (через компоненты Langflow).
4. **Tool backend:** Python + FastAPI (Mock API и endpoint для bug ticket).
5. **Интеграции:** REST API, webhook-эмуляция (например, Trello/Jira mock).
6. **Наблюдаемость:** структурированные JSON-логи + trace_id.

## 5. Архитектура (Router / Dispatcher)

### 5.1 Компоненты

| Компонент | Ответственность | Вход | Выход |
| --- | --- | --- | --- |
| Dispatcher Agent | Классифицирует обращение | Текст пользователя | `INFO`/`STATUS`/`BUG` + confidence + reason |
| RAG Agent | Отвечает только по базе знаний | Текст + top-k чанков | Ответ + ссылки на источники |
| Tool Agent | Проверяет статус заказа через API | Текст запроса | Статус заказа + пользовательский ответ |
| Triage Agent | Формирует баг-тикет | Текст + опц. логи/шаги | JSON тикета + результат POST |

### 5.2 Поток обработки

1. Входящее сообщение получает `Dispatcher Agent`.
2. Router переводит запрос в одну из веток.
3. Ветка обрабатывает запрос и возвращает итоговый ответ.
4. Каждый шаг пишет лог с `trace_id`, `agent`, `latency_ms`, `decision`.

## 6. Контракты данных (I/O Schemas)

### 6.1 Вход запроса

```json
{
  "trace_id": "uuid",
  "session_id": "string",
  "thread_id": "string",
  "user_id": "string",
  "message": "string",
  "timestamp": "ISO-8601"
}
```

### 6.2 Ответ Dispatcher

```json
{
  "category": "INFO | STATUS | BUG",
  "confidence": 0.0,
  "reason": "string"
}
```

### 6.3 Ответ INFO

```json
{
  "answer": "string",
  "sources": [
    {
      "doc_id": "string",
      "chunk_id": "string"
    }
  ],
  "grounded": true
}
```

### 6.4 Ответ STATUS

```json
{
  "order_id": "string",
  "raw_status": "processing | shipped | delivered | canceled",
  "answer": "string"
}
```

### 6.5 Ответ BUG

```json
{
  "title": "string",
  "description": "string",
  "priority": "High | Medium | Low",
  "repro_steps": "string",
  "expected": "string",
  "actual": "string"
}
```

### 6.6 Ответ ESCALATE

```json
{
  "action": "ESCALATE",
  "reason_code": "LOW_CONFIDENCE | POLICY_RISK | NO_KB_HIT | TOOL_FAILURE",
  "reason": "string",
  "handoff_payload": {
    "summary": "string",
    "collected_context": "string"
  }
}
```

## 7. Функциональные требования и критерии приемки

## 7.1 Routing (Dispatcher)

1. Модель возвращает только один label из `INFO`/`STATUS`/`BUG`.
2. При низкой уверенности (`confidence < 0.55`) система задает уточняющий вопрос.
3. Langflow Conditional Router направляет в нужную ветку.

**Acceptance Criteria:**

1. На тест-сете из 100 размеченных запросов accuracy >= 90%.
2. Нет невалидных label в output.

## 7.2 INFO (RAG)

1. Загрузка документов KB (правила возврата, доставка, оплата).
2. Chunking и эмбеддинг документов.
3. Семантический поиск top-k и генерация ответа строго по контексту.
4. Ответ сопровождается ссылкой/ID источника.

**Acceptance Criteria:**
1. >= 85% ответов содержат корректный релевантный источник.
2. В ручной проверке 0 критичных галлюцинаций на golden-наборе (минимум 30 кейсов).

## 7.3 STATUS (Tool Agent)

1. Извлечение `order_id` из текста.
2. Вызов `GET /order/{order_id}`.
3. Преобразование технического JSON в понятный ответ пользователю.

**Acceptance Criteria:**

1. Tool call success rate >= 95%.
2. Если `order_id` не найден, агент запрашивает уточнение.
3. Если API недоступен, пользователь получает fallback-ответ и предложение повторить позже.

## 7.4 BUG (Triage Agent)

1. Если данных недостаточно, агент задает уточняющие вопросы (ошибка, шаги, ожидание/факт).
2. Формирует структуру баг-тикета.
3. Выполняет POST в mock endpoint или выводит JSON для ручной отправки.

**Acceptance Criteria:**

1. 100% тикетов содержат `title`, `description`, `priority`.
2. >= 80% тикетов в ручной оценке считаются "готовыми к передаче разработке без доуточнений".

## 8. Нефункциональные требования

1. **Latency:** p95 полного ответа <= 5 сек для mock-интеграций.
2. **Надежность:** graceful fallback при ошибках LLM/API.
3. **Безопасность:** маскирование потенциальных PII в логах.
4. **Наблюдаемость:** JSON-логи по каждому шагу + trace_id.
5. **Повторяемость:** фиксированные промпты и версия тест-сета для регрессии.

## 9. Prompt Engineering и guardrails

1. Использовать role-prompting и few-shot для Dispatcher.
2. Для RAG-agent добавить жесткое правило: отвечать только по найденному контексту.
3. Для Tool-agent: обязательная проверка наличия `order_id` до вызова API.
4. Для Triage-agent: шаблон заполнения тикета с обязательными полями.
5. Добавить запрет на chain-of-thought в пользовательский ответ; наружу отдается только финальный формат.

### 9.1 Защита от Prompt Injection

1. Игнорировать пользовательские инструкции, конфликтующие с системными правилами и политиками.
2. Никогда не раскрывать системные промпты, внутренние политики, токены и секреты.
3. Разрешать tool-calling только по allowlist инструментов и только в ожидаемом формате аргументов.
4. При признаках инъекции (например, "игнорируй предыдущие инструкции") возвращать безопасный ответ и `ESCALATE`.

### 9.2 Пример системного промпта Dispatcher

> Ты маршрутизатор запросов поддержки.  
> Верни строго одно значение: INFO, STATUS или BUG.  
> INFO — правила, возврат, доставка, условия.  
> STATUS — запрос о заказе и его местонахождении/сроке, обычно с номером.  
> BUG — техническая ошибка, сбой, код ошибки, некорректная работа интерфейса.  
> Никакого дополнительного текста.

## 10. Обработка ошибок и fallback-политика

1. **Low confidence routing:** запросить уточнение.
2. **RAG no-hit:** сообщить, что данных в базе нет, и предложить эскалацию на оператора.
3. **Tool timeout/error:** вежливый fallback + повторная попытка позже.
4. **Invalid bug payload:** задать недостающие вопросы по шаблону.

## 11. План тестирования и evaluation

### 11.1 Набор тестов

1. **Routing tests:** минимум 100 размеченных сообщений.
2. **RAG tests:** минимум 30 вопросов по KB + проверка источников.
3. **STATUS tests:** минимум 25 кейсов (валидный id, отсутствует id, несуществующий id, timeout).
4. **BUG tests:** минимум 25 кейсов (полное/неполное описание, приоритизация).

### 11.2 Метрики

1. Routing accuracy.
2. Grounded answer rate.
3. Tool success rate.
4. Ticket completeness rate.
5. End-to-end latency p50/p95.

## 12. Этапы реализации (roadmap)

1. **Sprint 1:** базовый Langflow-граф, Dispatcher, mock API, минимальные промпты.
2. **Sprint 2:** полноценный RAG и BUG triage, fallback-логика, логирование.
3. **Sprint 3:** evaluation pipeline, улучшение метрик, polish для демо.

## 13. Артефакты для портфолио

1. Репозиторий с README и архитектурной схемой.
2. Примеры входов/выходов по каждой ветке.
3. Таблица метрик до/после улучшений промптов.
4. Короткое демо-видео (3–5 минут) с 6–8 сценариями.
5. Раздел "Ограничения и следующие шаги" (честный инженерный разбор).

## 14. Риски и mitigation

1. **Нестабильная классификация:** расширить few-shot и golden set.
2. **Галлюцинации в INFO:** усиливать grounded-ограничения и проверку источников.
3. **Хрупкость извлечения order_id:** добавить regex + LLM hybrid extraction.
4. **Шумные баг-репорты:** обязательные уточняющие вопросы перед отправкой.

---

## Definition of Done (MVP)

MVP считается завершенным, если:

1. Реализованы и интегрированы все 3 ветки (`INFO`, `STATUS`, `BUG`) через Dispatcher.
2. Выполнены целевые quality gates:
   - Routing accuracy >= 90%
   - Tool success rate >= 95%
   - Ticket completeness rate = 100%
3. Подготовлен демо-набор кейсов и README с архитектурой, метриками и ограничениями.
